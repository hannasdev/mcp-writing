The problem

Context windows don't scale to novels.

A 90,000-word manuscript is ~120,000 tokens. Paste your whole book into a prompt and you've already burned most of a context window — before asking a single question.

mcp-writing solves this with a metadata-first design. The AI queries a structured SQLite index to find the relevant scenes, characters, and beats. Then it loads only the prose it actually needs — cutting context usage by 95%+ on most queries.

The result: the AI can reason about a 100-scene manuscript as fluently as a short story.

Typical query: context consumption
Naive "paste manuscript" approach ~90%
mcp-writing: metadata query ~3%
+ targeted prose (2–3 scenes) ~8%
~11% total vs ~90% — same answer, 8× cheaper. Metadata queries never load prose at all.
How it works

Three layers, always in order.

Step 01
🗃️

Index, don't paste

On first run, sync walks your Scrivener sync folder and builds a SQLite index of every scene — title, characters, beat, POV, logline, word count, and tags. Re-runs are incremental.

Step 02
🔍

Query metadata first

find_scenes, get_arc, and search_metadata answer structural questions from the index alone — no prose loaded, no tokens wasted. Full-text search runs in milliseconds against 100+ scenes.

Step 03
✏️

Load prose surgically

When the AI needs to read or edit prose, get_scene_prose loads only the specific scenes it asks for. Every proposed edit goes through a two-step confirm workflow — diff first, commit on approval.

Scenarios

What you can actually do with it.

Real workflows writers use with mcp-writing every day.

Goal

Understand whether a character's 55-scene presence earns its space

A major character appears in 55 scenes across the full manuscript. You want to know if the arc justifies that footprint — or if it's bloated — without rereading every scene to find out.

  1. Ask the AI to run get_arc for the character across the full project.
  2. The result is paginated: metadata for all 55 scenes, ordered by timeline, grouped by part — no prose loaded.
  3. The AI reads the beat sequence and identifies structural jobs each part is doing.
  4. Stale scenes are surfaced automatically. Load only those with get_scene_prose to verify.

Goal

Catch timeline contradictions before beta readers see them

Before sending chapters to beta readers or an editor, run a targeted continuity pass on a specific character thread, subplot tag, or date range — without rereading the whole book.

  1. Run find_scenes filtered by tag (e.g. injury, promise) or character.
  2. Use get_arc to review the thread in timeline order.
  3. Load only the suspect scenes with get_scene_prose for close reading.
  4. Attach continuity notes with flag_scene — they stay on the scene for your next revision pass.

Goal

Let AI propose line edits — you approve before anything is saved

No silent rewrites. Every AI edit goes through a two-step confirm workflow: propose a diff, review it, then commit or discard. Full git history means you can always roll back to any previous version.

  1. Ask the AI to tighten, rewrite, or revise a specific scene.
  2. It calls propose_edit and shows you the diff.
  3. You review the changes — accept with commit_edit or discard entirely.
  4. Every committed edit creates a git commit. Use list_snapshots to roll back.

Goal

Add a new character without breaking pacing

You have a character concept — but you're not sure where they fit in a 100-scene manuscript without crowding existing arcs or disrupting the structural beats you've already built.

  1. Ask the AI to map structural insertion windows using find_scenes on the full project.
  2. Pull existing character arcs with get_arc to understand what the new character can't overlap.
  3. The AI proposes a minimal beat structure (intro, fracture, exit) with exact scene placement recommendations.
  4. Review and update the character sheet with update_character_sheet to anchor the plan.

Features

Everything a novelist needs from an AI layer.

🗂️

Scrivener sync

Import from Scrivener External Folder Sync. Binder-ID identity means scenes stay stable even when you reorder them.

⚡

Full-text search

FTS5-powered search across scene titles and loglines. Find every scene mentioning a theme, prop, or phrase in milliseconds.

📊

Beat & arc queries

Filter by Save the Cat beat, character, chapter, part, POV, or tag. Trace any character's ordered journey across the whole manuscript.

🔒

Human-in-the-loop edits

All prose changes go through propose → diff → commit. Nothing is written to disk without explicit approval. Every commit is git-backed.

🏷️

Sidecar metadata

Metadata lives in .meta.yaml sidecar files alongside prose. Scrivener owns the .md files; MCP owns the metadata. No conflicts.

🔄

Staleness detection

When prose is rewritten, affected scenes are flagged as stale. Re-derive metadata with enrich_scene so your index stays accurate.

🌍

World entities

Structured character and place sheets with supporting notes. Query relationships, track mention counts, and keep world-building consistent.

📦

Review bundles

Generate structured review packages for a scene or chapter range — AI-assisted analysis, flagged issues, and editorial summaries in one pass.

🐳

Docker-ready

First-class Docker Compose support. Runs alongside OpenClaw or any MCP-capable AI gateway. Configurable via environment variables.

Start in under five minutes.

mcp-writing runs as a local HTTP server. Connect it to any MCP-capable AI gateway — Claude Desktop, OpenClaw, or your own tooling.
